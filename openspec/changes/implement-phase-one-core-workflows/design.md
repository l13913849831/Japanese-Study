## Context

第一阶段框架已经完成三类基础能力：

- 后端已经接入 Spring Boot、Flyway、JPA、统一响应和模块骨架。
- 前端已经接入 React、Vite、路由、统一 API client 和模块页面壳层。
- 数据库已经通过 `V1-V5` 落下词库、学习计划、卡片、模板、导出任务等表结构与默认模板种子。

当前缺失的是真正把这些资产串成完整业务流程的实现层。根据 `docs/system-usage-guide.md` 的下一步建议，第一阶段需要优先补齐五个核心闭环：词条导入、学习计划创建编辑、卡片预生成、模板预览、导出创建下载。这些能力彼此依赖，适合放在一个 change 中统一推进。

## Goals / Non-Goals

**Goals:**

- 让用户能够从词库导入词条，并在前端查看词条列表。
- 让用户能够创建和编辑学习计划，并触发卡片预生成。
- 让卡片查询接口返回真实的今日学习数据和日历聚合数据。
- 让模板在保存前可以预览渲染结果，降低导出配置错误。
- 让导出任务可以被创建、查询并下载实际文件。

**Non-Goals:**

- 本次不引入动态记忆算法，如 SM-2 或 FSRS。
- 本次不实现多用户、权限控制或任务队列系统。
- 本次不直接生成 `.apkg` 文件，只覆盖 CSV/TSV 与 Markdown。
- 本次不重构已有框架骨架，只在其基础上补齐业务实现。

## Decisions

### 1. 词条导入采用同步 CSV 解析 + 行级错误返回

词条导入先采用后端同步处理，单次覆盖第一阶段预期规模，并返回成功数量、跳过数量与行级错误信息。

- Why: 当前系统是单用户、第一阶段规模可控，同步方案足够简单且便于定位问题。
- Alternative considered: 用异步任务导入。该方案更适合大批量数据，但会增加状态管理和前端轮询复杂度。

### 2. 学习计划创建时执行“先校验、后生成、同事务落库”

学习计划在创建或需要重建时，先校验词库、模板、复习间隔与状态，再在同一事务内生成计划记录和对应卡片实例。

- Why: 计划与卡片实例强一致，分开提交容易留下半成品计划。
- Alternative considered: 计划保存后异步生成卡片。该方案扩展性更高，但第一阶段会引入状态不一致和额外恢复逻辑。

### 3. 卡片查询基于预生成实例，不在查询时实时计算

`today` 和 `calendar` 接口统一从 `card_instance` 读取，并按 `due_date`、`card_type`、`status` 聚合返回。

- Why: 数据库里已经有合适的结构和索引，查询路径更清晰，也方便后续记录复习状态。
- Alternative considered: 查询时按计划与词条动态推导。该方案减少存储，但会让查询逻辑和导出逻辑重复实现。

### 4. 模板预览统一收敛为“模板内容 + 示例上下文”的服务

模板预览不依赖真实学习计划，统一接收模板主体和示例变量上下文，后端返回渲染结果，前端据此展示预览。

- Why: 这样既能覆盖新建模板，也能覆盖编辑已有模板，不被真实业务数据阻塞。
- Alternative considered: 只允许对已有模板做数据库读取预览。该方案对新建流程不友好。

### 5. 导出先采用同步文件生成 + 数据库存储记录

导出请求先同步生成文件到约定目录，写入 `export_job` 记录，再通过下载接口返回文件流。

- Why: 第一阶段导出量不大，同步方案更直接，也便于排查。
- Alternative considered: 完整的后台任务队列。该方案更通用，但超出当前阶段复杂度目标。

## Risks / Trade-offs

- [同步导入和同步导出可能拉长请求时间] → 通过控制第一阶段数据规模，并在接口层限制单次输入范围。
- [计划重建可能导致旧卡片实例冲突] → 明确重建规则，必要时删除旧实例后全量重算，并保证事务一致性。
- [模板渲染可能被非法变量或模板语法击穿] → 维护白名单变量和统一异常映射，避免直接暴露底层错误。
- [前端页面在能力补齐后容易产生表单和表格复杂度上升] → 继续沿用模块分层和共享组件，避免页面内堆积业务逻辑。

## Migration Plan

1. 先补齐词库词条导入和词条查询，为学习计划提供稳定输入。
2. 再实现学习计划创建/编辑与卡片预生成，打通核心学习链路。
3. 补齐模板预览，降低导出前的配置不确定性。
4. 最后实现导出创建与下载，把学习结果输出链路补完整。

Rollback strategy:

- 如果某项能力未完成，可保留数据库基线和框架代码，仅回退对应模块实现。
- 对新增接口与页面入口采用增量方式，不修改既有健康检查和基础壳层行为。

## Open Questions

- 导出文件是统一落在本地磁盘目录，还是直接以内存流方式返回并只保留元数据记录。
- 学习计划编辑后是否总是全量重建卡片，还是只允许在特定状态下修改关键字段。
- CSV 导入是否需要支持覆盖模式，还是第一阶段只保留“新增 + 跳过重复”的模式。
