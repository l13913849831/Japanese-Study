# brainstorm: sync and productization extension

## Goal

在最小用户归属底座已经建立后，继续规划多端同步、备份恢复和更完整产品化能力的演进方向。

## What I already know

* 当前项目还没有账户体系，因此同步能力无法直接成立。
* 一旦 `P0-a` 建立，后续就会自然衍生：
  * 多端同步
  * 备份恢复
  * 更完整用户设置
  * 数据迁移与资料管理

## Requirements (evolving)

* 需要明确 `P0-b` 依赖 `P0-a` 完成之后再启动。
* 需要把同步、备份、恢复、设置这类能力整理成一个后续容器任务。
* 需要避免把 `P0-b` 提前混进当前主路径优化任务。
* 需要承接管理层能力的长期归属，例如导出历史、资料迁移、备份恢复和账户设置。

## Acceptance Criteria (evolving)

* [ ] 能说明 `P0-b` 的能力边界。
* [ ] 能说明它为什么不应和 `P0-a` 混做。
* [ ] 能为后续继续拆任务保留清晰入口。

## Out of Scope

* 这轮不做同步协议设计
* 这轮不做云端架构设计
* 这轮不做共享市场或协作能力

## Relationship To P1 Tasks

`P1-A / P1-B / P1-C` 负责学习主路径、完成反馈和复盘体验。

`P0-b` 负责更偏产品化和管理层的能力容器：

* 多端同步
* 备份 / 恢复
* 更完整账户设置
* 数据迁移
* 历史记录管理

学习闭环里的轻量导出动作并入 `P1-B`；导出历史与长期资料管理可在 `P0-b` 或后续管理任务中展开。
