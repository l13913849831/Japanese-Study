# brainstorm: sync and productization extension

## Goal

在最小用户归属底座已经建立后，继续规划多端同步、备份恢复和更完整产品化能力的演进方向。

## What I already know

* `P0-a` 已经落地，不再是待设计状态。
* 当前系统已经有：
  * 本地注册 / 登录 / 登出
  * 后端 session / cookie 鉴权
  * 内部 `user_account / user_identity / user_setting` 模型
* 当前文档 `docs/system-usage-guide.md` 已明确：
  * 当前仍然不包含多端同步
  * 当前仍然不包含 Keycloak / 微信登录
* 用户归属底座已经覆盖：
  * `word_set` 的 `scope + owner_user_id`
  * `anki_template / md_template` 的 `scope + owner_user_id`
  * `study_plan.user_id`
  * 知识点线的 `note_source + note` 拆层
* 现有导出能力已经可用，但仍是本地文件导出：
  * 导出文件写入 `APP_EXPORT_BASE_DIR`
  * `export_job` 记录仍围绕学习计划和文件路径
  * 还没有“账号级备份集 / 恢复点 / 跨设备同步状态”模型
* 一旦 `P0-a` 建立，后续就会自然衍生：
  * 多端同步
  * 备份恢复
  * 更完整用户设置
  * 数据迁移与资料管理

## Requirements (evolving)

* 需要明确 `P0-b` 依赖 `P0-a` 完成之后再启动。
* 需要把同步、备份、恢复、设置这类能力整理成一个后续容器任务。
* 需要避免把 `P0-b` 提前混进当前主路径优化任务。
* 需要承接管理层能力的长期归属，例如导出历史、资料迁移、备份恢复和账户设置。
* 当前已确定首个方向是“整体备份”，而不是分拆成资料备份和进度备份。

## Acceptance Criteria (evolving)

* [ ] 能说明 `P0-b` 的能力边界。
* [ ] 能说明它为什么不应和 `P0-a` 混做。
* [ ] 能为后续继续拆任务保留清晰入口。
* [ ] 能给出 `P0-b` 的首个 MVP 切入方向。

## Out of Scope

* 这轮不做同步协议设计
* 这轮不做云端架构设计
* 这轮不做共享市场或协作能力

## Relationship To P1 Tasks

`P1-A / P1-B / P1-C` 负责学习主路径、完成反馈和复盘体验。

`P0-b` 负责更偏产品化和管理层的能力容器：

* 多端同步
* 备份 / 恢复
* 更完整账户设置
* 数据迁移
* 历史记录管理

学习闭环里的轻量导出动作并入 `P1-B`；导出历史与长期资料管理可在 `P0-b` 或后续管理任务中展开。

## Repo Constraints

* 后端当前是 Spring Boot + PostgreSQL + Flyway，持久化结构变更必须走 migration。
* 当前认证实现是简单 session/cookie，适合作为后续“同一账号多设备”演进的起点，但还没有 token / device / sync cursor 概念。
* 当前导出文件落在服务端本地目录，更像“导出结果下载”，不是“用户资产备份”。
* 现有导出历史虽然已经按用户可见范围收口，但数据模型仍然依附 `study_plan`，不适合作为通用资料管理中心。

## Research Notes

### What current code suggests

* 账号底座已经足够支撑“同一用户在多个会话下访问同一份数据”这个方向。
* 真正缺的不是登录，而是：
  * 资料资产边界
  * 可恢复的备份单元
  * 跨设备一致性的同步语义
* 如果直接做“真同步”，会立刻牵涉：
  * 冲突处理
  * 增量同步协议
  * 设备状态与重试
  * 恢复后的幂等与审计
* 如果先做“账号级备份/恢复”，则更贴近当前导出与文件落盘能力，也更容易先形成产品化闭环。

### Feasible approaches here

**Approach A: 先做账号级备份/恢复中心** (Recommended)

* How it works:
  * 先把“可导出学习结果”升级为“可生成、列出、下载、导入的账号备份集”
  * 备份单元面向用户资产，而不是单个学习计划
  * 恢复先支持“导入到当前账号”或“覆盖指定范围”
* Pros:
  * 最贴近当前本地文件导出实现
  * 可以先补产品化资产管理，而不提前引入同步协议复杂度
  * 为以后做云端同步提供数据包与恢复语义
* Cons:
  * 不能立刻解决多端实时一致性
  * 需要先定义备份覆盖范围和恢复冲突策略

**Approach B: 先做多设备会话一致性底座**

* How it works:
  * 围绕当前 session 登录补“设备 / 会话 / 最近同步时间 / 最近活动”管理
  * 先让账号管理更完整，再为后续同步铺路
* Pros:
  * 先把“这是一个多端产品”最基础的账户管理感做出来
  * 风险低于直接上同步协议
* Cons:
  * 对用户学习资料可迁移、可恢复的价值不够直接
  * 很容易变成纯账户管理补丁，离真正 `P0-b` 核心偏远

**Approach C: 直接定义同步协议和增量游标**

* How it works:
  * 先抽象各类学习数据的变更流、版本号和冲突规则
  * 再补客户端同步入口
* Pros:
  * 最直接通向“多端同步”
* Cons:
  * 对当前仓库阶段过重
  * 会把大量架构不确定性提前拉进来

## Current Recommendation

`P0-b` 的第一个 MVP 更适合走 **Approach A**：

* 先做“账号级备份 / 恢复 / 资产管理”容器
* 把导出历史从“学习计划附属物”逐步提升为“用户资料资产”
* 后续再从备份快照、恢复记录、资料版本这些概念，平滑过渡到真正同步

## Current Delivered Slice

当前这一轮已经完成的 `P0-b` 首个交付切片：

* 账号整体备份包格式
* 当前账号整体备份导出下载
* 恢复前安全快照
* 覆盖恢复链路
* 前端 `/backups` 页面与导航入口

这意味着：

* **本地备份 / 本地恢复 MVP 已完成**
* `P0-b` 当前剩余的更高阶方向转为：
  * 云端托管备份
  * VIP 备份历史
  * 多设备/会话治理
  * 更完整产品化资料管理

## User Decision

### Chosen MVP Direction

用户已选择：**整体备份**

含义：

* 第一版备份单元不是“单个计划导出”
* 也不是“资料和进度拆开分别操作”
* 而是以“当前账号的一份完整学习资产快照”为核心

### Chosen Restore Strategy

用户已选择：**整包覆盖恢复**

含义：

* 恢复目标是“把当前账号恢复到某个备份快照”
* 恢复时优先覆盖学习资产，而不是做逐项合并
* 恢复不应直接改写认证凭据或系统默认资料

由此推出的第一版恢复语义：

* 覆盖对象：
  * 当前账号下的用户私有内容资产
  * 当前账号下的学习进度数据
  * 当前账号设置
* 默认不覆盖：
  * `SYSTEM` scope 默认资料
  * `user_identity` / 登录凭据
  * 其他用户的数据
* 安全前提：
  * 恢复前应能提示“这会覆盖当前账号现有学习资产”
  * 最稳妥的 MVP 做法是恢复前自动生成一份当前状态备份，作为回滚点

## Derived Backup Scope

基于当前仓库结构，整体备份的候选范围可先按下面理解：

* 用户资料与偏好
  * `user_account` 的可迁移资料字段
  * `user_setting`
* 用户私有内容资产
  * `word_set(scope=USER)` + `word_entry`
  * `anki_template(scope=USER)`
  * `md_template(scope=USER)`
  * `note_source(scope=USER)`
* 用户学习进度
  * `study_plan`
  * `card_instance`
  * `review_log`
  * `note`
  * `note_review_log`
* 用户资产历史
  * 备份任务自身的元数据
  * 现有 `export_job` 是否纳入，需要单独决策

默认不建议直接纳入第一版整体备份的内容：

* `SYSTEM` scope 的默认资料
  * 这些属于系统种子数据，不应跟着个人备份重复携带
* `user_identity.password_hash`
  * 更像认证凭据，不是学习资产
  * 直接恢复凭据会把账户恢复和学习数据恢复耦合得过深
* 既有导出产物文件本身
  * 现有文件更接近一次性导出结果，而不是结构化备份资产

## Derived Product Shape

结合当前代码现状，`P0-b` 第一版更像一个新的“账号资产中心”，而不是扩展现有复盘导出页：

* 现有 `/export-jobs`：
  * 面向单个学习计划
  * 面向某个日期的复盘材料
  * 表单字段是 `planId + exportType + targetDate`
* 整体备份 / 覆盖恢复：
  * 面向当前账号
  * 面向完整学习资产快照
  * 需要备份列表、下载、恢复、恢复风险提示、恢复前安全快照

因此当前倾向：

* 不把第一版整体备份硬塞进 `export_job` 模型
* 更适合单独抽“backup / restore”领域对象与页面入口

### Chosen Entry Shape

用户已选择：**独立“备份与恢复”页面**

含义：

* 第一版不是挂在 `/account` 的附属区块
* 也不是只做后端接口、没有前端入口
* 而是作为管理区中的独立页面存在

## Derived UI Direction

结合当前前端路由和菜单结构，第一版更适合：

* 新增独立路由，例如 `/backups`
* 在管理区菜单中新增独立入口
* 保留现有 `/export-jobs`
  * 它继续承担“计划复盘材料导出”
* 新页面承担：
  * 创建账号整体备份
  * 查看备份历史
  * 下载备份包
  * 执行覆盖恢复
  * 展示恢复风险与确认提示

## Open Product Decision

当前还需要明确恢复执行前的安全策略。

推荐方向：

* 恢复前自动生成一份“当前状态备份”作为回滚点
* 再执行覆盖恢复

原因：

* 覆盖恢复本身是高风险动作
* 自动安全快照能显著降低误操作成本
* 也更符合“备份与恢复中心”的产品心智

## New Open Question

当前新增一个更上游的产品/架构问题：

* 备份包最终存放在哪里？

候选方向：

* 存在当前业务数据库里
* 存到对象存储，例如 S3
* 作为可下载 / 可上传的导出文件

这个决策会直接影响：

* MVP 的基础设施复杂度
* 灾备真实性
* 多端同步的后续演进路径

### Chosen Backup Storage Direction

用户已选择：

* **第一版先做导出文件型备份**
* **同时为后续接入 S3/对象存储预留接口**
* **未来可能把托管备份做成 VIP 功能**

由此推出的产品方向：

* MVP 阶段：
  * 备份本体是结构化导出文件
  * 用户可以下载保存
  * 恢复时通过上传备份文件执行
* 演进阶段：
  * 在不改变备份包格式的前提下，增加“托管存储”
  * 托管后端可以先抽象为对象存储接口，再由本地文件 / S3 分别实现
* 商业化阶段：
* 免费用户保留“导出到本地文件 / 手动上传恢复”
* VIP 用户增加“云端托管备份 / 更长历史 / 一键恢复”

### Chosen Free Backup History Shape

用户已选择：**免费用户不保留服务端备份历史**

含义：

* 免费用户点击“创建备份”后，目标是直接下载备份压缩包
* 免费用户的 `MANUAL_BACKUP` 不进入服务端长期历史列表
* 服务端可以为了生成下载响应短暂落临时文件，但返回后应删除

由此推出的 MVP 形态：

* 免费用户：
  * 创建整体备份
  * 立即下载备份压缩包
  * 通过上传本地备份文件执行恢复
  * 不提供长期托管的 `MANUAL_BACKUP` 历史列表
* VIP 用户：
  * 才开放服务端 / 对象存储托管历史
  * 才有真正意义上的历史列表、下载、按历史一键恢复

## Derived Architecture Direction

第一版就应避免把“备份领域”直接写死到本地磁盘路径。

推荐抽象：

* `BackupPackageBuilder`
  * 负责把当前账号学习资产序列化成统一备份包
* `BackupStorage`
  * 负责保存 / 读取备份包
  * 第一版实现为 `LocalFileBackupStorage`
  * 后续可扩为 `S3BackupStorage`
* `BackupRestoreService`
  * 负责上传恢复、覆盖恢复、安全快照
* `BackupPolicyService`
* 负责决定当前用户是否允许使用“托管备份”等高级能力

这样做的价值：

* 第一版仍然可以快速落在本地文件
* 但控制层、领域层、前端页面都不需要感知底层未来是不是 S3
* 也方便后续把“托管备份”作为 VIP 能力开关，而不是重做备份协议

## Derived MVP Scope

当前 `P0-b` 第一版可以先收敛为以下范围：

* 新领域：
  * `backup`
* 新页面：
  * `/backups`
* 第一版能力：
  * 为当前账号生成整体备份文件
  * 上传备份文件执行覆盖恢复

* 第一版不做：
  * S3 真接入
  * VIP 权限链路落地

### Chosen Safety Snapshot Policy

用户已选择：**恢复前自动安全快照与用户主动备份分开处理**

含义：

* `SAFETY_SNAPSHOT` 不应和用户主动创建的 `MANUAL_BACKUP` 混为一类
* 即便免费用户不保留服务端 `MANUAL_BACKUP` 历史，系统仍可为恢复安全单独处理 `SAFETY_SNAPSHOT`
* `SAFETY_SNAPSHOT` 更像恢复过程中的保护机制，而不是正式长期存档

### Chosen Free Restore Safety Flow

用户已选择：**免费用户恢复前，系统先生成安全快照并立即下载到本地，再继续恢复**

含义：

* 免费版不保留服务端长期 `MANUAL_BACKUP`
* 免费版也不保留服务端长期 `SAFETY_SNAPSHOT`
* 恢复前的安全保护通过“先下载一份回滚快照”实现

推荐流程：

1. 用户上传本地备份压缩包准备恢复
2. 系统先基于当前账号状态生成一份 `SAFETY_SNAPSHOT`
3. 浏览器先下载这份安全快照
4. 用户确认后，系统再继续执行覆盖恢复
5. 服务端生成过程中的临时文件在流程结束后删除

### Chosen Safety Snapshot Retention

用户已选择：**`SAFETY_SNAPSHOT` 只保留最近 3 条**

含义：

* 免费版当前已改为“恢复前先下载安全快照”，因此不依赖服务端长期保留
* “最近 3 条”的规则主要保留给后续 VIP 托管模式或需要短期服务端托管的场景

## Current Recommendation

关于“免费用户是否也可使用云端托管备份”，当前更推荐：

* **免费用户可使用备份功能**
  * 手动创建整体备份
  * 下载备份文件到本地
  * 上传本地备份文件恢复
* **免费用户不开放真正的云端托管备份**
  * 不把备份包持久化到 S3/对象存储
  * 不承诺跨实例、跨部署的托管保留
* **VIP 用户再开放云端托管能力**
  * 备份包上传到 S3/对象存储
  * 更长历史保留
  * 一键恢复
  * 后续可扩自动备份

原因：

* 如果免费版一开始就直接给真正的云端托管，后续 VIP 价值会被削弱
* 当前阶段先把“备份格式、恢复语义、历史规则”做稳，比先做付费托管更重要
* 这样第一版仍然对免费用户有真实价值，但不会提前锁死商业化空间

## Clarified File Retention Policy

需要明确区分两类文件：

### 1. 服务端托管的备份历史文件

这类文件属于系统内“已登记的备份历史”：

* 用户主动创建的 `MANUAL_BACKUP`
* 系统恢复前自动生成的 `SAFETY_SNAPSHOT`

它们会：

* 在有托管能力时出现在 `/backups` 历史列表
* 暂存在服务端本地磁盘或对象存储
* 受数量上限和清理策略约束

### 2. 用户上传用于恢复的本地备份文件

这类文件默认不应被当成“新的服务端备份历史”长期保留。

推荐第一版规则：

* 上传文件仅作为“恢复输入”
* 服务端可先落到临时文件或临时目录做校验 / 解包 / 恢复
* 恢复成功或失败后，临时上传文件应被删除
* 它默认不进入 `/backups` 历史列表

原因：

* 否则“上传恢复一次”就会额外产生一份长期存储副本
* 容易导致磁盘膨胀
* 也会让“历史备份”和“外部导入文件”语义混淆

## Disk Growth Control

为避免本地磁盘快速膨胀，第一版应至少同时具备这些控制：

* 免费用户的 `MANUAL_BACKUP` 不做服务端长期保存
* 上传恢复文件只做临时保存，处理后删除
* 免费用户的 `SAFETY_SNAPSHOT` 生成后直接下载，不做服务端长期保存
* 删除历史时同步删除对应物理文件
* 需要有“元数据存在但物理文件缺失”的容错与清理逻辑

### Chosen Backup Package Format

用户已选择：**备份包做成压缩包**

当前倾向：

* 第一版使用单文件压缩包，例如 `zip`
* 包内至少包含：
  * `manifest.json`
  * 用户学习资产数据文件
  * 版本号 / 生成时间 / 备份类型等元信息

这样做的价值：

* 本地下载和上传恢复都更自然
* 后续切到 S3 时不需要改变备份包协议
* 更方便做版本校验、完整性校验和未来扩展

## Clarified Backup Type Semantics

### `MANUAL_BACKUP`

`MANUAL_BACKUP` 指：

* 用户主动点击“创建备份”生成的一份完整账号学习资产压缩包
* 它属于用户可见、可下载、可用于后续恢复的正式备份
* 在免费版里，它默认是“生成后立即下载”，而不是长期托管在服务端

用户心智上可以把它理解成：

* “我手动存了一份完整存档”

### `SAFETY_SNAPSHOT`

`SAFETY_SNAPSHOT` 指：

* 系统在执行“覆盖恢复”前，自动为当前账号生成的一份保护性快照
* 它的主要目的不是长期归档，而是为了恢复失败或恢复后反悔时可以回滚

用户心智上可以把它理解成：

* “系统在帮我做危险操作前，先偷偷垫了一层安全网”

## Clarified User Flows

### 免费用户正常使用流程

1. 用户进入 `/backups`
2. 点击“创建备份”
3. 系统生成一个 `MANUAL_BACKUP` 压缩包
4. 浏览器直接下载这个压缩包
5. 用户把它保存在自己本地

### 免费用户恢复流程

1. 用户在本地保留过一个备份压缩包
2. 用户在 `/backups` 上传这个压缩包
3. 系统先校验格式与版本
4. 系统基于当前账号状态生成一份 `SAFETY_SNAPSHOT`
5. 浏览器先下载这份安全快照，作为回滚文件
6. 用户确认后，系统使用上传文件执行覆盖恢复
7. 上传文件和生成过程中的临时文件处理完成后删除，不默认进入长期历史列表

### 后续 VIP 用户流程

VIP 用户在上面免费流程基础上，额外增加“托管备份”能力：

1. 用户点击“创建备份”
2. 系统仍然先生成同一种备份压缩包
3. 存储层不再只写本地磁盘，还可以写入 S3 / 对象存储
4. 用户在任何接入该账号的实例里，都可以看到托管备份历史
5. 用户可以直接基于托管备份做一键恢复，而不依赖自己本地先保存文件

VIP 的增量价值主要不在“备份格式”，而在：

* 托管保存
* 更长历史
* 更稳定的跨设备可取回
* 后续可扩自动备份

## Finalized MVP Boundary

基于当前已确认的产品决策，`P0-b` 第一版 MVP 边界收敛为：

* 面向对象：
  * 免费用户
* 交付目标：
  * 提供账号整体备份压缩包导出
  * 提供基于本地备份压缩包的覆盖恢复
  * 在恢复前自动生成并下载一份安全快照
* 不包含：
  * 免费用户服务端长期备份历史
  * S3 / 对象存储真接入
  * VIP 权限控制真正落地
  * 自动备份
  * 跨设备托管恢复

换句话说，第一版先交付：

* “我可以把自己当前账号完整存档到本地”
* “我可以用本地存档恢复”
* “恢复前系统会先给我一份后悔药”

而不交付：

* “系统永久帮我保存这些备份”

## Refined Acceptance Criteria

* [ ] `/backups` 页面可访问，并能清晰区分“创建备份”和“上传恢复”。
* [ ] 免费用户点击“创建备份”后，浏览器会直接下载一个整体备份 `zip`。
* [ ] 该 `zip` 至少包含 `manifest.json` 和当前账号学习资产数据。
* [ ] 免费用户上传备份 `zip` 后，系统会先生成并下载 `SAFETY_SNAPSHOT`，再执行恢复确认。
* [ ] 覆盖恢复只作用于当前账号学习资产，不改写登录凭据和系统默认资料。
* [ ] 上传恢复所用的临时文件在流程结束后被删除。
* [ ] 当前实现已抽出 `BackupStorage` 一类存储接口，第一版可用本地文件实现，后续可切 S3。
* [ ] 当前实现已预留策略判断入口，后续可为 VIP 托管备份接入能力开关。
