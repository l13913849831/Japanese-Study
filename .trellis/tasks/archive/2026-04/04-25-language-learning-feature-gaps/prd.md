# brainstorm: language learning feature gaps

## Goal

基于当前仓库已经落地的真实能力，判断这个项目如果继续往“语言学习软件”走，而不只是单词/知识点管理工具，还缺哪些更像学习产品核心的功能，并把候选方向压缩成一版可排优先级的建议。

## What I already know

* 当前项目已经是一个可运行的单机学习 MVP，不是最早的纯准备工具。
* 当前已有两条学习主线：
  * 单词学习：词库、导入、学习计划、今日卡片、复习历史、仪表盘、导出
  * 知识点学习：知识点 CRUD、Markdown 导入预览、FSRS 复习、知识点仪表盘
* 当前前端主路由：
  * `/dashboard`
  * `/word-sets`
  * `/study-plans`
  * `/cards`
  * `/notes`
  * `/notes/review`
  * `/notes/dashboard`
  * `/templates`
  * `/export-jobs`
* 当前后端主模块：
  * `wordset`
  * `studyplan`
  * `card`
  * `dashboard`
  * `note`
  * `template`
  * `exportjob`
  * `shared`
* 当前数据库明确是 PostgreSQL，不是 MySQL。
* 当前产品还不包含：
  * 多用户
  * 登录鉴权
  * 用户账户模型
  * token / session 认证链路

## Research Notes

### What similar language-learning products usually include

* 语言学习产品通常不只提供卡片复习，还会覆盖：
  * 学习目标与分级
  * 听说读写至少两项以上训练
  * 长周期反馈
  * 错题/薄弱点回流
  * 动机维持机制
* 单纯“词库 + 复习”更接近学习引擎或个人工具，而不是完整语言学习产品。

### Constraints from our repo/project

* 当前已有最稳的资产是：
  * 词条数据
  * 学习计划
  * 复习日志
  * 知识点复习
* 当前还没有账号体系和云同步，所以涉及跨端和社交的能力都不适合立刻做。
* 当前前端是管理台风格，继续加重内容录入页会比补主学习体验更容易，但产品收益更低。

### Feasible approaches here

**Approach A: 先把“学习闭环”做完整** (Recommended)

* How it works:
  * 优先补错题回流、薄弱项回顾、阶段目标、学习完成反馈
* Pros:
  * 最贴近现有数据资产
  * 能快速把现有功能变成更完整的学习循环
* Cons:
  * 对“内容更丰富”这类需求提升有限

**Approach B: 往“综合语言训练”扩**

* How it works:
  * 增加听力、例句音频、跟读、阅读材料或语法专题
* Pros:
  * 更像完整语言学习软件
  * 用户感知会更强
* Cons:
  * 研发跨度大
  * 现有仓库基础对这类能力支撑较弱

**Approach C: 先补产品基础设施**

* How it works:
  * 增加账户、同步、设置、数据备份、模板共享
* Pros:
  * 为长期产品化做准备
* Cons:
  * 对当前“学得更好”帮助最弱

## Priority Recommendation

### P0-a: 最小用户归属底座

**目标**

为“每个用户有自己的学习进度、自己的资料、自己的默认学习偏好，同时又能使用系统默认资料”提供最小可行底座。

**当前缺口**

* 没有用户表 / 账户表。
* 没有登录能力和用户身份边界。
* 核心学习数据没有 `user_id` / `owner` 归属。
* 没有用户级设置，无法把“默认先单词还是先知识点”变成真正的用户偏好。
* 还没有定义“系统默认资料”和“用户私有资料”的并存规则。

**为什么先做**

* 一旦主路径开始按用户偏好决定，偏好就应尽快从本地状态升级为用户设置。
* 一旦允许用户上传自己的资料，同时也允许使用默认资料，数据归属问题就不适合继续往后拖。
* 单词线后补成本还可控，但知识点线当前把内容和进度绑得较紧，越晚补用户归属，后续拆分代价越大。

**2026-04-28 当前进展**

* 第一批 `P0-a` 已完成：
  * 后端已有 `user_account / user_identity / user_setting`
  * 已接本地登录 + session cookie
  * 已提供 `/api/auth/login`、`/api/auth/logout`、`/api/me`、`/api/me/settings`
  * 单词线、模板线、学习计划、导出、单词 dashboard/weak-items 已收口到当前用户
  * 前端已有登录页、鉴权守卫、Header 偏好切换

**当前剩余缺口**

* 知识点线仍未完成用户归属拆层：
  * `note` 仍然同时承载内容和进度
  * 还不支持“系统默认知识点资料 + 每个用户独立进度”
* 当前只实现 `LOCAL` 身份提供方：
  * `KEYCLOAK`
  * `WECHAT_MP`
  仍是预留位，未接入实际认证流程
* 还没有注册、账户管理、修改默认密码、多账号运营能力
* 还没有做真实浏览器登录态 + 实库迁移的人工冒烟验证

### P1-A: 统一学习主路径

**目标**

让用户每天打开后，不需要自己判断“先去哪一页、做什么”，而是根据用户偏好和当日进度，直接进入当前最该完成的学习动作。

**当前缺口**

* 工作台虽然已经统一，但还没有把“下一步动作”收成真正的单一路径。
* 单词线和知识点线仍然需要用户自己决定先做哪条、做完一条后是否继续另一条。
* 会话完成后的衔接还不够强，缺少“完成当前 -> 自动过渡到下一个任务”的表达。
* 当前缺少真正的用户级顺序偏好，因此第一版主路径应建立在 `P0-a` 提供的用户设置之上。

**这一层不包括**

* 不涉及重训练模块
* 不要求先建立更完整的学习资料结构

### P1-B: 闭环呈现与完成反馈

**目标**

让单词线和知识点线不只是“机制上已经闭环”，而是把这个闭环明确呈现给用户：开始、完成、剩余、恢复、毕业、下一步都能被看见和感知。

**当前缺口**

* `Again -> 回捞 -> WEAK -> 再练` 这条机制已经存在，但页面对这条链路的表达还不够完整。
* 还缺更明确的“今天完成了什么、还剩什么、为什么会进入薄弱轮”的反馈。
* 还缺更稳定的“完成当前会话后下一步怎么办”的系统引导。
* 还缺对“恢复成功 / 从薄弱项毕业 / 今日任务已清空”这类节点的更清晰呈现。

**内容组织与关联在这里的位置**

* 单词与知识点的关联、按标签/主题回看、相近内容整理，确实会增强这一层。
* 但这些能力依赖学习资料本身足够完整，不能单靠“学习软件壳层”自动成立。
* 所以它更适合作为这一层的配套增强，而不是单独列成当前最高优先级。

### P1-C: 长期反馈与复盘

**目标**

让用户不只知道“今天学了没有”，还知道“最近有没有在进步、哪里在积压、哪些内容长期偏弱”。

**当前缺口**

* 缺少周/月尺度的学习反馈。
* 缺少按标签、类型、来源、计划维度的长期弱点归纳。
* 缺少“新学 / 复习 / 弱项恢复”的结构化占比反馈。

**内容组织与关联在这里的位置**

* 更细的复盘能力会受资料质量影响。
* 如果标签体系、词和知识点的关联、本体分类还不够稳定，长期复盘的精度也会受限。
* 因此这部分也更适合作为 `P1-C` 的配套深化，而不是先单独立项。

### P0-b: 同步与产品化延伸

**目标**

在 `P0-a` 已完成最小用户归属底座后，再补多端同步、备份恢复和更完整产品化能力。

**当前缺口**

* 多端同步 / 备份 / 恢复
* 更完整账户设置
* 产品化层的分享、迁移、恢复能力

**为什么放后面**

* 它的重要性高，但前提是先有用户归属底座。
* 对当前“先把单词学习 + 知识点学习做成真正可持续使用的软件”来说，不如 `P0-a + P1` 组合直接。

**为什么先放 P2**

* 这些能力重要，但当前不会直接改善“单词学习 + 知识点学习”的每日主动作。
* 在当前单机 MVP 阶段，先补主路径、闭环和反馈，收益更直接。

## Candidate Feature Design: Daily Weak-Item Recovery Loop

### Goal

把 `Again` 从“只影响下次正式复习时间”的评分，升级成“同时影响今天会话内补救”的动作，让用户在当天就能再见一次没掌握的内容，并在必要时把内容沉淀为“易错项 / 薄弱项”。

### Problem

当前系统已经有：

* 评分
* 复习日志
* 下次复习排期
* 基础掌握度状态

当前系统还没有：

* 当天会话内回捞
* 薄弱项升级规则
* 今日末尾专项再练
* 易错项列表
* 易错项自动毕业规则

### Design Summary

推荐采用“两层机制”：

1. **会话内即时补救**
   * 第一次 `Again`：放回今日队列尾部，今天稍后再见一次
2. **会话外薄弱项沉淀**
   * 第二次 `Again` 或达到薄弱阈值：标记为 `WEAK`
   * 今日主队列结束后，允许用户选择“薄弱项再练一轮”
   * 同时进入“易错词 / 易错知识点”列表，后续可手动查看

### Why this is preferred

相比“只做错题本页面”或“只改长期排期”，这套方案更平衡：

* 当天体验自然，不打断主流程
* 有真正补救动作，不只是多一个标签
* 可以保留一个单独列表做提醒和回顾
* 能兼容单词线和知识点线

### Core Rules

#### 1. 今日队列内规则

* 第一次 `Again`
  * 当前项从“已评分”变成“待今日重做”
  * 放回当前队列尾部
  * 不立即弹出，不打断当前节奏
* 第二次 `Again`
  * 仍放回今日薄弱队列
  * 同时标记为 `WEAK`
* `Hard / Good / Easy`
  * 正常完成当前项本轮处理
  * 继续影响长期排期

#### 2. 薄弱项规则

建议 MVP 规则先简单，不直接上复杂统计模型：

* 满足任一条件即视为 `WEAK`
  * 同一天内两次 `Again`
  * 最近 N 次正式复习里 Again 占比过高
* `WEAK` 是一种显式状态，不等于永久错题本成员

#### 3. 今日收尾规则

当今日主队列完成后：

* 如果存在 `WEAK` 项或今日多次 `Again` 项
* 展示一个收尾选择：
  * `再练一轮薄弱项`
  * `今天先结束`

这一步不强迫用户，但显式提醒。

#### 4. 易错项列表规则

系统维护一个“易错词 / 易错知识点”列表：

* 进入条件：
  * 达到 `WEAK` 阈值
* 退出条件：
  * 后续正式复习出现 `Good` 或 `Easy`
  * 或者用户手动移除

这意味着它既是提醒集合，也是可清理集合。

### Data Model Direction

MVP 不建议先做独立“错题本实体”，建议先做轻量状态扩展。

#### Word line

在现有 card / review 体系上补：

* 今日会话态：
  * `sessionAgainCount`
  * `sessionQueueState`
* 长期弱项态：
  * `weakFlag`
  * `weakMarkedAt`
  * `weakSource`

#### Note line

在现有 note / note_review_log / FSRS 体系上补：

* 今日会话态：
  * `sessionAgainCount`
  * `sessionQueueState`
* 长期弱项态：
  * `weakFlag`
  * `weakMarkedAt`

说明：

* 会话态更适合短生命周期，不一定最终落长期表
* `weakFlag` 才是需要跨天保留的核心字段

### UX Flow

#### Main review session

```text
[今日主队列]
   ↓
用户看到当前项
   ↓
打 Again
   ↓
当前项进入队尾
   ↓
继续做后面的项
   ↓
稍后再次看到该项
   ↓
再次 Again -> 标记 WEAK
   ↓
主队列结束
   ↓
提示：是否再练一轮薄弱项
```

#### Weak-item list

```text
[易错项列表]
   ↓
按单词 / 知识点分组
   ↓
显示加入原因、最近错误时间、最近表现
   ↓
用户可手动查看或移除
   ↓
正式复习出现 Good/Easy 时自动毕业
```

### Scope Recommendation

#### MVP

* `Again` 后回今日队尾
* 第二次 `Again` 标记 `WEAK`
* 今日结束时可选“薄弱项再练一轮”
* dashboard 增加“当前易错项数量”摘要
* 提供基础易错项列表页，并在 dashboard / review 页给入口
* `Good/Easy` 自动从易错项中移除

#### Later

* 基于最近多次表现的更稳定薄弱项判定
* 区分“短期卡壳”和“长期薄弱”
* 易错项专项训练模式
* 近义词 / 混淆词联动回顾

### Trade-offs

#### Why not only use long-term scheduling

因为只改下次正式复习时间，用户当天感知不到“补救”，学习闭环还是断的。

#### Why not only create a wrong-answer notebook

因为只建列表不改主流程，用户还是得额外主动进去处理，太容易沦为摆设。

#### Why not force weak-item extra round every day

因为会增加疲劳感。MVP 更适合做成可选收尾动作。

### Recommendation

推荐你刚才描述的方向，但稍微收紧成这一版：

* 第一次 `Again`：回今日队尾
* 第二次 `Again`：标记 `WEAK`
* 今日结束：提示是否加练薄弱项
* 同时沉淀进易错项列表
* 后续正式复习出现 `Good` 或 `Easy`：自动毕业

这比“只回队尾”完整，也比“先做独立错题本系统”更自然。

### Naming decision

* 底层统一状态概念：`WEAK`
* 前台文案分开：
  * 单词线：`易错词`
  * 知识点线：`易错知识点` 或 `薄弱知识点`
* 两条线各自存储，不共用一张表

## Page Entry Recommendation

建议补一个独立入口，而不是只放在复习页内部。

### Entry shape

* dashboard 上增加：
  * `易错词`
  * `易错知识点`
  * 数量摘要
* 左侧导航增加一个聚合入口：
  * `/weak-items`
* 页面内分两个标签页：
  * `易错词`
  * `易错知识点`

### Why this shape

* 比拆成两个新一级路由更克制
* 比只在 review 页里藏一个侧栏更容易被看见
* 后续如果要加筛选、毕业记录、专项训练，也更容易扩

## Note-line Adaptation

### Conclusion

知识点线也适用“今日补救 + 易错项沉淀”这套思路，但不能照搬单词线。

### Why it is different

* 单词通常更短，回队尾多见几次成本低
* 知识点内容更长，重复太快会更累
* 知识点线已经接了 FSRS，长期排期本来就更细

### Recommended note-line behavior

#### Shared with word line

* `Again` 不只影响长期排期
* 需要有当天补救动作
* 需要有 `WEAK` 沉淀和易错知识点列表
* `Good/Easy` 可以自动毕业

#### Different from word line

* 知识点线不建议无限回队尾
* 更适合“当天最多回捞一次”
* 更适合在会话末尾做“薄弱知识点再练一轮”，而不是在主队列里来回穿插过多次

### Recommended note-line MVP

* 第一次 `Again`
  * 不立刻再次弹出
  * 放入“今日待回顾知识点”子队列
* 第二次 `Again`
  * 标记 `WEAK`
  * 进入易错知识点列表
* 今日主队列结束后
  * 可选择进入“薄弱知识点再练一轮”

### Why not mirror the word line exactly

如果知识点也像单词一样多次回队尾，用户会更容易疲劳，因为：

* 知识点每项阅读量更大
* 复习动作更偏理解和回忆，不只是识别
* 高频回捞更容易打断当天节奏

### Practical recommendation

因此建议：

* 单词线：允许更明显的回队尾
* 知识点线：保留一次当天回捞，但把重点放到“末尾再练一轮”和“易错知识点列表”

这样两条线保持同一理念，但细节强度不同。

### ADR-lite update

**Context**:

单词线和知识点线都需要“当天补救 + 薄弱项沉淀”，但用户希望前台文案符合各自语义，同时不想把底层概念做成两套互不兼容的机制。

**Decision**:

统一底层状态概念为 `WEAK`，前台按学习线分别展示为“易错词”和“易错知识点/薄弱知识点”，实现上各自落在本线的数据模型里，不混表。

**Consequences**:

* 前台表达更自然
* 后端规则可复用同一套判定思路
* 不会因为共用表而混淆单词与知识点
