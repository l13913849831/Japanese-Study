# brainstorm: infrastructure foundation gaps

## Goal

记录当前产品从单机学习 MVP 走向更稳定产品化所缺的基础设施能力，并明确把“最小用户归属底座”提升到当前最高优先级。

## What I already know

* 当前项目仍然是单机单用户形态。
* 当前没有：
  * 登录 / 注册
  * 用户模型
  * 数据隔离
  * token / session 鉴权链路
  * 多端同步 / 备份恢复
* 如果后续产品方向明确为：
  * 每个用户维护自己的学习进度
  * 每个用户可上传自己的学习资料
  * 同时又能使用系统默认资料
  那基础设施不能只理解成“晚点再做登录页”，而需要先补最小用户归属模型。

## Requirements (evolving)

* 需要把基础设施缺口记进 roadmap。
* 需要明确这些能力为什么重要。
* 需要区分：
  * `P0-a：最小用户归属底座`
  * `P0-b：同步 / 备份 / 多端等延伸能力`

## Acceptance Criteria (evolving)

* [ ] 能说明当前缺哪些基础设施能力。
* [ ] 能说明这些能力和当前学习主路径优化的前后依赖关系。
* [ ] 能为后续单独立项保留清晰入口。
* [ ] 能说明“默认资料”和“用户私有资料”未来如何并存。

## Out of Scope

* 这轮不直接实现登录或同步
* 这轮不进入权限模型设计细节

## Suggested Split

### P0-a: 最小用户归属底座

* 用户表 / 账户表
* 最基本登录能力
* 核心学习数据的 `owner` / `user_id`
* 用户设置（例如默认先单词还是先知识点）
* “系统默认资料” 与 “用户私有资料” 的并存规则

### P0-b: 同步与产品化延伸

* 多端同步
* 备份 / 恢复
* 更完整的账户设置
* 未来的分享、协作或模板市场能力
