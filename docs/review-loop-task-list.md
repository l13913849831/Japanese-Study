# 复习闭环任务清单

## 1. 任务目标

为当前系统补齐“卡片复习提交 -> 记录复习结果 -> 更新卡片状态 -> 查询复习历史”的最小闭环。

完成后，系统应从“学习计划与导出工具”升级为“可实际执行学习任务的系统”。

---

## 2. 范围边界

本任务聚焦最小可用闭环，不包含以下内容：

- FSRS / SM-2 等高级调度算法
- 复杂学习统计与仪表盘
- 多端同步
- 音频、发音、例句朗读

本任务只覆盖：

- 单卡复习提交
- 复习记录保存
- 卡片状态更新
- 单卡复习历史查询
- 前端 cards 页面复习操作与结果展示

---

## 3. 后端任务

- [ ] 3.1 设计并确认 review 相关 API 合同，包括请求体、响应体、错误码和状态变更规则
- [ ] 3.2 在 `card` 模块新增复习提交接口：`POST /api/cards/{cardId}/review`
- [ ] 3.3 在 `card` 模块新增复习历史查询接口：`GET /api/cards/{cardId}/reviews`
- [ ] 3.4 实现 `review_log` 写入逻辑，保存 `rating`、`responseTimeMs`、`note`、`reviewedAt`
- [ ] 3.5 为复习提交增加业务校验，例如卡片存在性、是否允许重复提交、状态是否合法
- [ ] 3.6 在复习成功后更新 `card_instance.status`，并明确与“今日卡片查询”的联动行为
- [ ] 3.7 为复习接口补齐统一错误响应，保持与现有 `ApiResponse` / `BusinessException` 风格一致
- [ ] 3.8 补充必要的 DTO、Service、Repository 调整，保持 `card` 模块边界清晰

---

## 4. 前端任务

- [ ] 4.1 在 `cards` 页面增加卡片复习操作区，支持 `AGAIN`、`HARD`、`GOOD`、`EASY`
- [ ] 4.2 为复习提交增加可选输入项，例如响应耗时和备注
- [ ] 4.3 提交成功后刷新今日卡片列表或当前卡片状态，确保页面反馈及时
- [ ] 4.4 在卡片页面展示最近一次复习结果
- [ ] 4.5 增加单卡复习历史展示区域，可先用简单表格或抽屉实现
- [ ] 4.6 处理复习接口加载态、错误态和成功提示，保持与现有 Ant Design + React Query 模式一致
- [ ] 4.7 保证 cards 页面在无计划、无卡片、提交失败时的状态展示清晰可用

---

## 5. 合同与数据一致性检查

- [ ] 5.1 确认 `review_log.rating` 与前端按钮枚举完全一致
- [ ] 5.2 确认卡片复习后 `card_instance.status` 的更新规则已在代码和文档中统一
- [ ] 5.3 确认今日卡片查询、卡片历史查询、复习提交三者的数据视图不冲突
- [ ] 5.4 确认错误码、字段名、时间格式与现有 API 规范一致

---

## 6. 验证任务

- [ ] 6.1 后端至少覆盖一条复习成功路径和一条非法提交路径
- [ ] 6.2 验证复习提交后能够新增 `review_log` 记录
- [ ] 6.3 验证复习提交后今日卡片状态会正确变化
- [ ] 6.4 验证复习历史查询可返回当前卡片的历史记录
- [ ] 6.5 验证前端 cards 页面可完成一次从查看卡片到提交复习再到看到反馈的完整流程

---

## 7. 完成标准

- [ ] 用户可以在系统内对单张卡片完成一次复习打分
- [ ] 系统会保存复习记录并可查询
- [ ] 卡片状态在前后端表现一致
- [ ] cards 页面具备最小可用的复习交互闭环
- [ ] 文档与实现保持一致，后续可直接基于该任务清单继续拆解

---

## 8. 后续可继续扩展的子任务

在本任务完成后，可以继续扩展：

- 复习统计与每日进度
- 连续学习天数
- 更细粒度的卡片状态流转
- 基于复习结果的高级调度算法
- 计划维度的学习完成率展示
